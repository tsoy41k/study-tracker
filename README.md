# Study Tracker

A Flask web application that lets a student log study sessions per subject
with a built-in timer, enter the grades they actually received, and get
ML-driven grade predictions and study recommendations.

## How the workflow looks for the student

1. **Register** with a numeric Student ID + password.
2. On first entry the student is sent to the **Setup** screen and chooses
   an *analysis period* (7 / 14 / 30 / 60 / 90 days).
3. The dashboard automatically contains 6 default subjects:
   - Internet of Things
   - Digital Marketing
   - System analysis and design
   - Distributed systems and cloud computing
   - Technology law
   - Business information systems project

   The student can add their own subjects or delete any of them.
4. During the analysis period the student:
   - picks a subject, starts the **timer**, presses **Stop & Save** at
     the end — a study record is written to the DB.
   - enters **grades** (0–100) they have received (quiz, midterm, etc.).
5. **Recommendations are locked** until the analysis period ends. The
   dashboard banner shows how many days are left.
6. After the period ends, the ML module produces recommendations based
   on the data collected during the period. Grade predictions are
   available as soon as the student enters at least one grade.

## How recommendations work (pt. 6 of the spec)

The recommendation engine lives in [`ml_model.py`](ml_model.py) and runs
on every call to `/api/recommendations`.

**Step 1 — analysis-period gate.** It first reads the user's analysis
period (start date + duration in days). If the current time is *before*
the end date, it returns a single info message ("Recommendations will
be generated in N day(s)") and stops. No advice is produced.

**Step 2 — per-subject aggregation.** Once the period has ended, the
engine computes for each subject:
- number of study sessions in the DB,
- total minutes studied,
- number of grades entered,
- average grade.

**Step 3 — rule application.** Recommendations are generated by a
small rule set, evaluated against those per-subject numbers:

| Rule | Trigger | Message type |
| ---- | ------- | ------------ |
| *No study* | A subject has 0 sessions during the period | `warning` |
| *Under-studied subject* | One subject has < 50% of the time of the most-studied one | `focus` |
| *Low average grade* | A subject's average grade < 60/100 | `warning` |
| *Time–grade mismatch (high time, low grade)* | ≥ 3 h studied but avg grade < 70 — suggests revising study method | `tip` |
| *Time–grade mismatch (low time, good grade)* | ≤ 30 min studied but avg grade ≥ 80 — current approach works well | `success` |
| *Balanced* | None of the above fires | `success` |

This is intentionally simple and explainable — every recommendation can
be traced back to a concrete number on the dashboard. The ML part
(linear regression) is used separately for the *predicted next grade*,
not for recommendations.

## How grade prediction works

For each subject:

1. Pull the student's grades for that subject from the DB.
2. Build training data:
   - `X[i]` = cumulative study minutes on this subject **at the moment
     the i-th grade was entered**
   - `y[i]` = the grade itself
3. Fit a `sklearn.linear_model.LinearRegression` on `(X, y)`.
4. Predict the grade for `X*` = the student's *current* total study
   minutes on this subject.
5. Clamp the result to the range `[0, 100]`.

Edge cases:
- 0 grades → `"no grades yet — add at least one grade to enable prediction"`.
- 1 grade → fall back to that grade (no regression possible).
- All grades entered with the same total study time → fall back to the average.

## Database schema

`study_tracker.db` (SQLite, auto-created) has four tables:

| Table          | Key columns |
| -------------- | ----------- |
| `users`        | `id` (numeric Student ID, PK), `password_hash`, `created_at`, `analysis_period_days`, `analysis_started_at` |
| `subjects`     | `id` (PK), `user_id` (FK), `name`, `created_at` — unique per user |
| `study_records`| `id` (PK), `user_id`, `subject_id`, `started_at`, `ended_at`, `duration_sec` |
| `grades`       | `id` (PK), `user_id`, `subject_id`, `grade` (0–100), `note`, `created_at` |

Foreign keys cascade on delete.

## API reference (JSON; all require login)

| Method | Path                          | Description                          |
| ------ | ----------------------------- | ------------------------------------ |
| GET    | `/api/subjects`               | List user's subjects                 |
| POST   | `/api/subjects`               | Add subject `{name}`                 |
| DELETE | `/api/subjects/<id>`          | Delete subject (and its data)        |
| POST   | `/api/records`                | Add session record                   |
| GET    | `/api/records[?subject_id=]`  | List records                         |
| POST   | `/api/grades`                 | Add grade `{subject_id, grade, note?}` |
| GET    | `/api/grades[?subject_id=]`   | List grades                          |
| GET    | `/api/predict/<subject_id>`   | Predicted next grade                 |
| GET    | `/api/recommendations`        | Recommendations + per-subject stats  |
| GET    | `/api/period`                 | Analysis period status               |

## Setup & run

Requires Python 3.10+ (tested on Python 3.14).

```powershell
pip install -r requirements.txt
python app.py
```

Open http://127.0.0.1:5000 — register an account, choose an analysis
period, then start tracking.

## Tests

```powershell
python test_app.py
```

## Notes & limitations

- The Flask `secret_key` in `app.py` is a placeholder; change it for any
  real deployment.
- Authentication is intentionally minimal (Student ID + password). No
  password reset, no rate limiting.
- Recommendations are rule-based on purpose: every message is fully
  explainable. The ML model (linear regression) is used for grade
  prediction only.
